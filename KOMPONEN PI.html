00 — Konsep Dasar

Konsep Dasar PI System

Sebelum masuk ke konfigurasi teknis, penting untuk memahami apa sebenarnya fungsi setiap komponen di PI System dan bagaimana mereka saling berhubungan. Section ini menjelaskan konsep PI Data Archive, PI Tags, dan PI AF dengan bahasa yang mudah dipahami.

Konsep 01 — PI Data Archive

Apa itu PI Data Archive?

PI Data Archive adalah database khusus time-series — yaitu database yang dirancang dari awal untuk menyimpan data yang berubah seiring waktu (nilai + timestamp). Berbeda dengan database relasional seperti SQL Server yang menyimpan baris-kolom statis, PI Data Archive menyimpan jutaan deret waktu dari ribuan sensor secara bersamaan dengan efisiensi tinggi.

Bayangkan sebuah pabrik dengan 10.000 sensor, masing-masing mengirim data setiap 1 detik = 10.000 nilai per detik = 864 juta data per hari. Database SQL biasa akan collapse dalam hitungan jam. PI Data Archive mampu menanganinya berkat dua mekanisme filter otomatis (Exception Reporting + Swinging Door Compression) yang mampu mereduksi data hingga 90% tanpa kehilangan akurasi.

Fungsi utama: menerima data real-time dari lapangan (via PI Interface), menyimpannya dengan kompresi lossless, dan menyajikan kembali — baik nilai saat ini maupun historis bertahun-tahun — dalam milidetik.

🏭 Dunia Nyata

Lemari arsip raksasa yang menyimpan rekaman pembacaan setiap alat ukur di pabrik — setiap menit, setiap hari, selama bertahun-tahun. Kamu bisa tanya: "Berapa tekanan kompresor 3 tahun lalu pada jam 2 siang?" dan dapat jawabannya dalam hitungan detik.

🖥️ PI Data Archive

Time-series database yang menyimpan nilai + timestamp dari setiap PI Tag. Dua layer: Snapshot (nilai terkini, di RAM) dan Archive (historis, di disk, file .arc). Data dikompres otomatis dengan algoritma Swinging Door.

Exception Reporting (Filter 1) — Data dari DCS hanya diteruskan ke Snapshot jika nilainya berubah lebih dari batas yang ditentukan (excdev). Kalau suhu stabil di 95°C, tidak perlu kirim 3.600 data yang sama dalam 1 jam.

Swinging Door Compression (Filter 2) — Nilai di Snapshot hanya ditulis ke Archive permanen jika tidak bisa direkonstruksi secara linear dari dua titik sebelumnya (batas compdev). Inilah yang membuat PI bisa menghemat 90% storage tanpa kehilangan akurasi.

ℹ️
PI Data Archive ≠ SQL Database PI Data Archive bukan relational database — tidak ada JOIN tabel atau arbitrary SQL. PI punya query language sendiri (PI Time Expression) dan diakses via PI SDK, PI Web API, atau PI OLEDB. Kelebihannya: untuk data time-series, PI jauh lebih cepat dan efisien dari SQL Server mana pun.
Konsep 02 — PI Tags / PI Points

Apa itu PI Tags (PI Points)?

PI Tag (resminya disebut PI Point) adalah unit penyimpanan terkecil di PI Data Archive. Satu PI Tag = satu deret waktu (time series) dari satu variabel proses. Setiap kali sensor membaca nilai, data tersebut masuk ke PI Tag yang sesuai lengkap dengan timestamp-nya.

Analoginya: kalau PI Data Archive adalah sebuah buku besar berisi ribuan halaman, maka setiap PI Tag adalah satu halaman khusus yang mencatat satu hal — misalnya halaman "Suhu Kompresor-01 Inlet". Halaman itu berisi deretan baris: jam berapa, berapa derajat, sampai bertahun-tahun ke belakang.

Setiap PI Tag punya atribut metadata — nama unik, satuan engineering, tipe data, batas range, aturan kompresi, dan kontrol keamanan. Jumlah PI Tag yang bisa dibuat dibatasi oleh lisensi PI Server (misal: lisensi 10.000 tags = maksimal 10.000 deret waktu bersamaan).

🌡️ Dunia Nyata

Setiap instrumen di pabrik (thermometer, pressure gauge, flow meter, vibration sensor) punya lembar log sendiri. Lembar itu punya nama (misal TI-101), satuan (°C), range (0–200), dan catatan pembacaan setiap menitnya.

🏷️ PI Tag

Satu PI Tag mewakili satu instrumen. Tag punya nama unik (COMP01.SUCT.PRESS), tipe data (Float32), satuan (psig), dan menyimpan seluruh historis nilai + timestamp dari instrumen tersebut.

Tiga Kondisi Nilai PI Tag yang Perlu Dipahami

Status Normal
Nilai Numerik

Tag menampilkan angka normal seperti 95.3 °C. Artinya data mengalir normal dari sumber ke PI Archive — semuanya bekerja dengan baik.

System Digital States
Pt Created / Shutdown / Comm Fail

Pt Created = tag baru, belum ada data. Shutdown = PI service mati. Comm Fail = koneksi ke sumber putus. No Data = tidak ada data dalam rentang waktu tersebut.

Digital Tag
ON/OFF · Running/Stopped

Tag bertipe Digital menampilkan text state bukan angka — misal Running, Stopped, Fault, Open, Closed sesuai Digital State Set yang dikonfigurasi.

Konsep 03 — PI Asset Framework (PI AF)

Apa itu PI Asset Framework?

PI AF adalah layer konteks yang memberikan makna pada data PI. Tanpa PI AF, PI Data Archive hanya menyimpan ribuan tag dengan nama seperti FIC-101, TIC-202 — angka-angka tanpa konteks, tanpa hubungan satu sama lain. Siapa yang tahu FIC-101 itu milik unit mana? Mengukur apa? Apa batas normalnya?

PI AF menjawab semua pertanyaan itu dengan membangun model hierarkis aset pabrik secara digital — dari level Site, Area, Unit, Equipment, hingga Measurement — dan menghubungkan setiap node ke PI Tag yang sesuai. PI AF juga memungkinkan kalkulasi derived values, event tracking, dan notifikasi alarm.

Konsep kunci: Element (objek fisik seperti Kompresor), Template (blueprint untuk semua objek sejenis), dan Attribute (properti dari objek, linked ke PI Tag). Hasilnya: satu "Compressor-01" di AF sudah punya semua datanya — suhu, tekanan, vibrasi — terorganisir rapi dan bisa diakses oleh siapa saja.

🏗️ Pabrik Fisik

Pabrik punya struktur: Site → Area Produksi → Unit Kompresi → Kompresor-01 → Sensor Suhu Inlet. Setiap engineer langsung tahu data itu dari mana dan artinya apa.

🌳 PI AF Hierarchy

AF mereplikasi struktur itu secara digital: FSO_SITE → GAS_COMP → Compressor-01 → Inlet_Temperature. Attribute Inlet_Temperature linked ke PI Tag COMP01.TEMP.IN di Data Archive.

💡
Hubungan Ketiga Komponen Utama PI PI Data Archive = Gudang penyimpanan data mentah.
PI Tags = Rak-rak berlabel di gudang, masing-masing menyimpan satu jenis data time-series.
PI AF = Peta dan katalog gudang — menjelaskan rak mana ada di zona mana, milik departemen apa, dan boleh diakses siapa.

2.2 Storage Architecture

📸

Snapshot Database

Menyimpan nilai terkini dari setiap PI tag. Tersimpan di RAM untuk akses ultra-cepat. Di-persist ke disk secara periodik. Service: pisnapss.

🗂️

Archive Database

Data historis dalam file .arc (data) + .ann (annotation). Satu file archive bisa menampung miliaran event. Service: piarchss.

piarch01.arc      # Archive sebelumnya (sudah di-shift)
piarch01.ann
piarch02.arc      # Archive lebih lama, dst...
piarch02.ann
⚠️
Archive Sizing — Jangan Abaikan! Archive terlalu kecil → shift terlalu sering → performa turun drastis.
Archive Size ≈ Tag Count × Data Rate × Retention (days) × 16 bytes / Compression Ratio

2.3 Compression & Exception Reporting

📉

Exception Reporting

Filter di sisi Interface. Data hanya dikirim ke Snapshot jika berubah > excdev atau sudah melewati excmax detik. Menghemat bandwidth jaringan.

🗜️

Swinging Door Compression

Filter di sisi Archive. Data Snapshot hanya ditulis ke .arc jika tidak bisa diinterpolasi linear dari dua titik sebelumnya (batas compdev). Menghemat disk.

💡
Analogi Swinging Door: Bayangkan dua pintu berputar di setiap data point. Selama data baru masih dalam "ayunan" kedua pintu (bisa diprediksi linear), data tidak disimpan. Hanya ketika ayunan dipaksa menutup oleh deviasi signifikan, data baru ditulis ke archive.

3.3 Cara Membuat PI Tags

Manual via PI SMT (Point Builder)

  1. Buka PI System Management Tool (PI SMT) → pilih PI Server
  2. Navigasi ke Points → Point Builder
  3. Klik New, isi: Tag Name, Point Type, Descriptor, Engineering Units
  4. Set Point Source ke kode interface (contoh: L untuk OPC)
  5. Isi Instrument Tag sesuai tagname di OPC server / DCS
  6. Atur nilai excdev, excmax, compdev, compmax sesuai kebutuhan
  7. Klik Apply — tag langsung aktif dan siap menerima data

4.3 PI AF Analytics

⚡

Expression Analysis

Rumus matematika dievaluasi real-time setiap ada data baru. Contoh: efisiensi, conversion, ratio.

📊

Rollup Analysis

Agregasi dari child ke parent: Total, Average, Min, Max. Ideal untuk KPI level site atau area.

🔔

Event Frame Generator

Buat Event Frame otomatis saat kondisi terpenuhi — misal suhu > 120°C selama > 5 menit berturut-turut.

-- PI AF Analytics Expression Examples

-- Compression Ratio
TagVal('Discharge_Pressure') / TagVal('Suction_Pressure')

-- Rata-rata suhu 1 jam terakhir
TagAvg('Inlet_Temperature', '*-1h', '*')

-- Flag alarm overtemperature
If TagVal('Inlet_Temperature') > 120 Then 1 Else 0

-- Total runtime bulan ini (dalam menit)
TagTot('Operating_Status', 'MonthStart', '*', 0, 1) / 60

5.2 PI Buffer Subsystem (PIBufss)

PIBufss menyimpan data sementara di disk lokal Interface Node ketika PI Data Archive tidak tersedia. Data di-replay otomatis saat koneksi pulih — zero data loss architecture.

7.1 PI Web API

RESTful API berbasis HTTP/JSON yang berjalan di atas IIS — memberikan akses ke Data Archive dan PI AF dari aplikasi modern, web, dan cloud.

EndpointMethodFungsi

/piwebapi/streams/{webId}/valueGETCurrent value (snapshot)
/piwebapi/streams/{webId}/recordedGETHistorical recorded data

import requests

BASE = "https://PISRVCISDEMO/piwebapi"
AUTH = ("piadmin", "password")

# Current value
web_id = "P0-abcXYZ..."
resp = requests.get(f"{BASE}/streams/{web_id}/value", auth=AUTH, verify=False)
val = resp.json()
print(f"{val['Timestamp']}: {val['Value']} {val['UnitsAbbreviation']}")

# Historis 1 jam terakhir
resp = requests.get(f"{BASE}/streams/{web_id}/recorded",
    params={"startTime": "*-1h", "endTime": "*"}, auth=AUTH, verify=False)
for item in resp.json()["Items"]:
    print(f"{item['Timestamp']}: {item['Value']}")

08 — Guidelines

Best Practices

8.1 Tag Naming Convention

📝
Format Rekomendasi: SITE-EQUIP-NUM.PARAMETER
Contoh: FSO-COMP-01.SUCT.PRESS, TAL-SEP-02.LVL.OIL
Gunakan titik (.) sebagai pemisah hierarki — konvensi paling umum di industri migas.
  • Gunakan ALL CAPS konsisten untuk semua tag name
  • Sertakan site prefix untuk deployment multi-site
  • Gunakan singkatan standar: TEMP, PRESS, FLOW, LVL, VIB, SPD, CURR, POWR
  • Hindari spasi dan karakter khusus (hanya titik dan underscore yang aman)
  • Dokumentasikan naming convention di PI Tag Dictionary sebelum implementasi dimulai

8.2 AF Design Principles

  • Maksimal 6–8 level kedalaman hierarchy untuk performa query yang optimal
  • Selalu gunakan Element Templates — jangan buat atribut manual berulang
  • Check In setiap selesai session editing — perubahan belum tersimpan sampai di-check-in
  • Satu AF Database per site / per plant yang logically independent
  • Test semua analytics di mode Preview sebelum diaktifkan ke production
