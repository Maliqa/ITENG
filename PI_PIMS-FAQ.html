Tag ini kenapa nilainya "Comm Fail" terus? Data tidak masuk sama sekali.
Operasi🔥 Hot Issue
▼
⚠️ Penyebab Umum

Comm Fail artinya PI Interface berhasil terhubung ke PI Server, tapi tidak berhasil membaca nilai dari sumber data (OPC Server / DCS ABB). Ini bukan masalah PI Archive-nya, tapi masalah di jalur komunikasi antara Interface dan sumber data.

✅ Langkah Solusi
1
Buka Matrikon OPC Explorer di Interface Node → connect ke ABB OPC Server → cek apakah item tersebut masih ada dan Quality-nya Good.
2
Cek atribut instrumenttag di PI SMT — pastikan nama OPC Item ID-nya sama persis dengan yang ada di ABB OPC Server (case-sensitive!).
3
Cek apakah instrumen fisik di lapangan dalam kondisi baik — Comm Fail bisa juga muncul kalau instrumen mati atau kabel terputus.
4
Restart PI Interface service: Restart-Service "PI-OPCInt-1" di PowerShell Interface Node.
5
Lihat log interface di C:\Program Files\PIPC\Interfaces\OPCInt\OPCInt1.log untuk detail error spesifik.
⚠️
Jangan Langsung Salahkan PI! Selalu test OPC koneksi dulu via Matrikon OPC Explorer. Kalau OPC client bisa baca data normal tapi PI masih Comm Fail, baru masalahnya ada di konfigurasi PI Tag atau PI Trust.
2
Nilai yang tampil di PI Vision berbeda dengan pembacaan di DCS. Mana yang benar?
Operasi
▼
⚠️ Penyebab Umum

Ada beberapa kemungkinan: (1) PI Vision menampilkan nilai interpolated bukan snapshot, (2) ada delay network antara DCS dan PI Interface, (3) Exception filter menyebabkan nilai terakhir yang disimpan berbeda dengan nilai terkini di DCS.

✅ Cara Membedakan
1
Cek timestamp nilai di PI Vision — apakah sama dengan waktu sekarang atau tertinggal beberapa menit?
2
Di PI SMT → Points → klik tag → tab Snapshot. Nilai Snapshot = nilai terbaru yang PI terima dari Interface. Jika Snapshot sama dengan DCS tapi PI Vision berbeda, masalahnya di display setting Vision (interpolated vs recorded).
3
Kalau Snapshot juga berbeda dari DCS, turunkan nilai excdev tag tersebut agar perubahan kecil tetap terkirim ke PI.
ℹ️
PI bukan pengganti DCS readout! PI adalah historian — tujuannya menyimpan tren data, bukan real-time HMI. Perbedaan kecil (terutama pada nilai yang berubah sangat cepat) adalah normal karena exception filter.
3
Kenapa grafik trend di PI Vision kelihatan "melompat-lompat" dan tidak smooth?
Operasi
▼
⚠️ Penjelasan

Ini adalah efek normal dari PI Compression (Swinging Door Algorithm). PI tidak menyimpan setiap data point — hanya menyimpan titik-titik "patahan" ketika perubahan tidak bisa lagi direpresentasikan secara linear. Sehingga saat ditampilkan, garis terlihat seperti lurus-lurus yang disambung.

✅ Solusi
1
Di PI Vision, ganti mode display trend dari Recorded ke Interpolated — PI akan mengisi gap antar titik secara linear, hasilnya lebih halus.
2
Jika membutuhkan resolusi lebih tinggi, turunkan nilai compdev di atribut tag (misalnya dari 0.5% ke 0.1%). Tapi ini akan menambah storage usage.
💡
Grafik stepping adalah tanda bahwa PI Compression bekerja dengan baik — data dikompres efisien tanpa kehilangan akurasi yang berarti. Ini fitur, bukan bug.

Apakah nama PI Tag harus sama persis dengan tag di P&ID atau DCS?
Engineering
▼
✅ Jawaban

Tidak harus sama, dan justru sebaiknya berbeda. PI Tag name adalah identifier di dalam PI System, sedangkan nama instrumen di P&ID/DCS disimpan di atribut instrumenttag.

Untuk Wayang Windu, kita menggunakan naming convention: WAWW.{LOOP_ID}.{PARAMETER} — contoh: WAWW.FIC101.PV. Ini lebih konsisten dan mudah di-query dari sistem lain.

Kalau range instrumen berubah setelah kalibrasi ulang, apakah data historis PI ikut berubah?
Engineering
▼
✅ Jawaban

Tidak. PI menyimpan nilai aktual dalam satuan engineering (EU), bukan persentase range. Ketika atribut zero dan span diubah, itu hanya mempengaruhi cara PI menampilkan dan menginterpretasikan data baru yang masuk — data historis yang sudah tersimpan tidak berubah.

Namun, jika instrumen sebelumnya dikalibrasi ulang tapi nilai raw-nya sudah salah masuk ke archive, data tersebut tidak bisa dikoreksi secara otomatis — harus di-overwrite manual via PI Web API atau PICONFIG.

💡
Best practice: tambahkan catatan di PI Event Frame setiap kali ada kalibrasi ulang instrumen, sehingga ada jejak audit kapan range berubah.

10
Attribute di PI AF menampilkan "No Data" padahal PI Tag-nya ada dan sudah ada datanya.
Engineering🔥 Hot Issue
▼
⚠️ Penyebab Paling Umum

String Builder tidak bisa me-resolve tagname karena nama Element di AF tidak cocok dengan bagian nama PI Tag. Contoh: Element bernama Compressor-01 tapi PI Tag-nya COMP-01.SUCT.PRESS — String Builder %Element%.SUCT.PRESS akan menghasilkan Compressor-01.SUCT.PRESS yang tidak ada.

✅ Langkah Diagnosa
1
Di PI System Explorer → klik attribute → kanan atas ada tombol Test Value → klik → lihat tagname yang dihasilkan String Builder.
2
Bandingkan tagname hasil resolusi dengan nama tag yang ada di PI SMT. Harus sama persis.
3
Solusi: sesuaikan nama Element AF dengan prefix yang dipakai di PI Tag, atau ubah template String Builder agar cocok.
# Contoh masalah:
Element name  : Compressor-01
String Builder: %Element%.SUCT.PRESS
Hasil resolusi: Compressor-01.SUCT.PRESS  ← tidak ada!
PI Tag actual : COMP01.SUCT.PRESS

# Solusi A — rename element:
Element name  : COMP01  ← ganti agar cocok

# Solusi B — ubah String Builder:
String Builder: COMP%Element:~4,2%.SUCT.PRESS  ← pakai substring

Berapa storage yang dibutuhkan untuk 5–10 tahun ke depan?
ITManajemen
▼
✅ Formula Kalkulasi

Gunakan formula standar OSIsoft:

# Formula:
Storage = Tags × DataRate × Days × 16 bytes / CompressionRatio × 1.2

# Contoh Wayang Windu (5.000 tags, 5 tahun):
= 5000 × 60 ev/tag/day × 1825 days × 16 bytes / 5 × 1.2
= ~5.3 GB untuk 5 tahun  ← sangat kecil karena compression!

# Tanpa compression (worst case):
= 5000 × 60 × 1825 × 16 × 1.2
= ~26.3 GB  ← masih kecil!
💡
PI sangat efisien untuk storage. Untuk 5.000 tag dengan 5 tahun retention, 100–200 GB sudah lebih dari cukup. Gunakan PI Storage Longevity Calculator yang sudah dibuat untuk kalkulasi lebih akurat.

PI Interface gagal connect ke OPC Server ABB — error "Access Denied" atau DCOM error.
IT🔥 Hot Issue
▼
⚠️ Penyebab

DCOM (Distributed COM) yang digunakan OPC DA membutuhkan konfigurasi Windows permission khusus. Ini adalah masalah paling sering di deployment OPC DA.

✅ Langkah Fix DCOM
1
Buka dcomcnfg di Interface Node → Component Services → Computers → My Computer → Properties → tab COM Security → Edit Limits → centang Allow Remote Activation dan Allow Remote Launch untuk user service account.
2
Buka Windows Firewall → Allow TCP port 135 (RPC Endpoint Mapper) dari Interface Node ke ABB OPC Node.
3
Test via Matrikon OPC Explorer dari Interface Node — bisa connect ke ABB OPC Server? Jika belum bisa dari OPC Explorer pun, berarti DCOM belum beres.
4
Jika OPC Explorer sudah bisa tapi PI masih gagal, cek PI Trust di PI SMT — pastikan IP Interface Node sudah didaftarkan.
# Allow DCOM port 135 via PowerShell:
netsh advfirewall firewall add rule name="DCOM RPC" ^
  protocol=TCP localport=135 dir=in action=allow

# Test koneksi ke ABB OPC node:
Test-NetConnection 192.168.x.x -Port 135

17
Kalau PI Server mati, berapa banyak data yang akan hilang?
ManajemenIT
▼
✅ Jawaban

Dengan PIBufss aktif: nol data hilang. PIBufss (PI Buffer Subsystem) menyimpan semua data di disk lokal Interface Node saat PI Server tidak tersedia. Begitu PI Server kembali online, semua data yang tertahan di-replay otomatis ke Archive.

Data hanya akan hilang jika: (1) PIBufss tidak dikonfigurasi, (2) disk lokal Interface Node juga penuh/rusak, atau (3) Interface Node ikut mati.

SkenarioData Loss
PI Server mati, PIBufss aktif di Interface NodeNol (0)
PI Server mati, PIBufss tidak aktifSemua data selama downtime
PI Server + Interface Node mati bersamaanSemua data selama downtime
PI Collective (redundant) — 2 serverNol (0)

Vibrasi

21
Apakah data waveform / spectrum bisa disimpan di PI, atau hanya data trend (RMS, Peak)?
Vibrasi
▼
✅ Jawaban

Data trend (RMS, Peak, 1X, 2X) disimpan sebagai PI Tag Float32 — ini yang paling umum dan ringan untuk storage.

Data waveform/spectrum bisa disimpan di PI menggunakan tipe Blob (Binary Large Object), tapi ini membutuhkan space storage yang jauh lebih besar dan konfigurasi khusus. Biasanya waveform disimpan di SETPOINT CMS sendiri, dan PI hanya menyimpan derived values-nya.

Tipe DataDisimpan DiPI Tag Type
RMS, Peak, 1X, 2X trendPI ArchiveFloat32
Overall vibration levelPI ArchiveFloat32
Full waveform bufferSETPOINT CMS (recommended)Blob (opsional)
FFT Spectrum snapshotSETPOINT CMS (recommended)Blob (opsional)

🔮 Dari ECG Inc. / Predict-IT Predict-IT
23
Apakah PI AF structure perlu disiapkan sebelum Predict-IT bisa berjalan?
Predict-ITEngineering
▼
✅ Jawaban

Ya, AF structure adalah prasyarat utama Predict-IT. Predict-IT dari ECG Inc. menggunakan PI AF SDK untuk membaca data — ia mencari asset hierarchy, element, dan attribute yang sudah terdefinisi di AF. Tanpa AF yang terstruktur, Predict-IT tidak bisa berjalan.

✅ Yang Perlu Disiapkan di PI AF
1
Element Hierarchy yang mencerminkan asset yang akan dipantau (Equipment → Sub-Equipment).
2
Attribute linked ke PI Tags untuk semua variabel yang dibutuhkan model Predict-IT (temperature, pressure, vibration, flow).
3
Minimal historical data — Predict-IT butuh data historis untuk training model similarity. Minimum 3–6 bulan data bersih sangat disarankan.
4
Data tidak boleh ada gap panjang atau banyak Comm Fail — data kualitas buruk akan menghasilkan model prediksi yang tidak akurat.
24
Kalau ada gap data atau banyak Comm Fail di archive, apakah model Predict-IT masih valid?
Predict-IT
▼
✅ Jawaban

Tergantung seberapa banyak dan panjang gap-nya. Predict-IT menggunakan similarity-based model — model dibangun dari pola data historis normal. Gap data kecil dan terisolasi (beberapa jam) masih bisa ditoleransi. Gap panjang (berhari-hari) atau gap yang terjadi berulang kali akan menurunkan akurasi model secara signifikan.

⚠️
Rekomendasi: Selesaikan semua masalah Comm Fail dan pastikan data mengalir bersih minimal 3 bulan sebelum Predict-IT digunakan untuk training model produksi. Data berkualitas buruk = model prediksi tidak bisa dipercaya.

✅ Urutan yang Benar
# STOP — urutan dari atas ke bawah:
Stop-Service "PIBufss"       # 1. Buffer dulu
Stop-Service "pieventss"    # 2. Events
Stop-Service "piarchss"     # 3. Archive
Stop-Service "pisnapss"     # 4. Snapshot
Stop-Service "pinetmgr"     # 5. Network manager terakhir
Start-Sleep 5

# START — urutan terbalik (dari bawah ke atas):
Start-Service "pinetmgr"    # 1. Network manager dulu
Start-Sleep 3
Start-Service "pisnapss"    # 2. Snapshot
Start-Service "piarchss"    # 3. Archive
Start-Service "pieventss"   # 4. Events
Start-Service "PIBufss"      # 5. Buffer terakhir

# Verifikasi semua running:
Get-Service "pi*","PIBufss" | Select-Object Name, Status | Format-Table
⚠️
Jangan pernah kill piarchss paksa tanpa stop service dulu — bisa menyebabkan archive file corrupt.
