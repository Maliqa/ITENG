01 — Exception Reporting

Exception Reporting

Exception Reporting adalah mekanisme filter di level PI Interface atau PI Buffer. Tujuannya sederhana: jangan kirim data ke PI Server jika nilainya tidak berubah secara signifikan. Ini adalah filter pertama dan paling fundamental dalam pipeline PI System.

💡
Analogi Exception Bayangkan sensor tekanan membaca 10.001 bar, 10.002 bar, 10.001 bar setiap detik. Tanpa exception reporting, PI Server kebanjiran data yang pada dasarnya tidak ada informasinya. Exception Reporting memfilter: "kirim data hanya jika berubah lebih dari 0.1 bar (ExcDev), atau sudah 600 detik sejak pengiriman terakhir (ExcMax)."

Cara Kerja Exception Reporting

1

Interface Membaca Raw Value

OPCInt membaca nilai dari ABB DCS sesuai scan rate. Misal setiap 1 detik, sensor tekanan mengirim nilai baru via OPC DA dari alamat IP 172.18.1.37.

2

Bandingkan dengan Nilai Sebelumnya

PI Interface membandingkan nilai baru dengan nilai terakhir yang dikirim ke PI Server. Jika selisihnya ≤ ExcDev, data tidak dikirim (di-drop).

3

Evaluasi ExcDev — Exception Deviation

Threshold perubahan minimum. Satuan bisa absolut (Eng Units) atau persen dari span. Di Wayang Windu default: excdev = 0.1 Eng Units untuk tag ABB.

4

Evaluasi ExcMax — Exception Maximum Time

Maksimal waktu "diam" sebelum paksa kirim. Meskipun nilai tidak berubah, data tetap dikirim setiap ExcMax detik — sebagai bukti sensor masih hidup. Default di Wayang Windu: excmax = 3600 detik (1 jam) untuk SETPOINT, dan dikonfigurasi berbeda untuk ABB.

5

Evaluasi ExcMin — Exception Minimum Time

Minimum interval antar pengiriman. Mencegah "banjir" data jika sensor berubah sangat cepat. Default: excmin = 0 (tidak ada pembatasan minimum).

6

Data Lolos → Kirim ke Snapshot

Jika data lulus exception check (berubah lebih dari ExcDev, atau waktu sudah melebihi ExcMax), data dikirim ke PI Server dan masuk ke Snapshot Subsystem.

02 — Snapshot Subsystem

Snapshot Subsystem

Snapshot adalah database in-memory PI Server yang menyimpan nilai terkini setiap PI Tag. Dikelola oleh service pisnapss.exe yang berjalan di IDWWDW25SCADA01. Snapshot adalah "jendela real-time" yang dilihat oleh PI Vision, PI ProcessBook, dan semua klien PI saat mereka menampilkan nilai sekarang.

⚡
Snapshot ≠ Archive Snapshot hanya menyimpan 1 nilai per tag — nilai terkini. Snapshot tidak persistent: jika PI Server di-restart, Snapshot dikosongkan. Data historis tersimpan di Archive. Snapshot adalah "RAM", Archive adalah "Hard Disk".

Struktur Snapshot

// SNAPSHOT SUBSYSTEM — IN-MEMORY STRUCTURE

→
pisnapss.exe Snapshot DB RAM · in-memory
↕

→

→

Snapshot Event vs Time-Based

📡
Event-Driven (Advise)

Snapshot diupdate setiap ada perubahan dari interface — push model. Digunakan tag ABB dan SETPOINT di Wayang Windu (Location3=1=advise). Lebih efisien, latensi rendah.

⏱️
Polled (Scan)

Snapshot diupdate berdasarkan scan class — pull model. Interface query nilai setiap N detik. Digunakan jika sumber tidak support push notification.

03 — Swingdoor Trending

Swingdoor Trending Algorithm

Swingdoor Trending adalah algoritma kompresi data time-series yang dikembangkan OSIsoft khusus untuk PI System. Ide utamanya: simpan hanya titik data yang dibutuhkan untuk merekonstruksi trend asli dengan akurasi yang ditentukan. Titik data yang bisa "diprediksi" dari interpolasi linear dua titik sebelumnya → tidak disimpan.

🚪
Mengapa Disebut "Swingdoor"? Bayangkan ada dua pintu (swinging doors) yang terbuka di sekitar setiap titik data baru. Selama titik baru masih berada "di antara kedua pintu" (dalam batas toleransi kompres dari trend yang ada), titik tersebut tidak disimpan. Pintu baru hanya dibuat ketika titik data keluar dari batas toleransi — menandakan trend telah berubah signifikan.

Cara Kerja Swingdoor

// SWINGDOOR — VISUAL CONCEPT
Nilai asli: 10.0 10.1 10.2 10.3 10.4 10.6 10.5 10.3 11.5 11.6 11.7 · · · · · · · · ↑ · · Tersimpan: ● ● ● ↑ ↑ ↑ titik awal keluar batas titik baru Interpolasi: ─────────────────────────────────────────────/──────────── CompDev: ═══════════════════════════════════════════ ← 11.5 melewati batas!
1

Titik Pertama Selalu Disimpan

Ketika tag pertama kali menerima data, nilai pertama langsung dikirim ke Archive sebagai titik acuan (anchor point).

2

Hitung Batas Toleransi (Swingdoor)

Dari titik terakhir yang disimpan, algoritma membuat "koridor toleransi" — sebuah rentang nilai yang diterima berdasarkan CompDev (compression deviation). Selama nilai baru masih masuk koridor ini (bisa dicapai dengan interpolasi linear), nilai tidak disimpan.

3

Cek CompDev — Compression Deviation

Jika nilai baru keluar dari koridor (artinya trend berubah dan tidak bisa direpresentasikan oleh interpolasi linear dalam batas CompDev), titik sebelumnya disimpan ke Archive, dan koridor baru dibuka. Default tag ABB Wayang Windu: compdev = 0.2 Eng Units.

4

Cek CompMax — Compression Maximum Time

Sama seperti ExcMax, CompMax memaksa penyimpanan meskipun nilai stabil — agar ada data yang bisa dikonfirmasi aktif. Default SETPOINT Wayang Windu: compmax = 3600 detik.

5

Data Lolos → Kirim ke piarchss

Titik data yang "keluar koridor" (atau mencapai CompMax) dikirim oleh pisnapss ke piarchss untuk disimpan permanen di file .arc di disk.

📉
Rasio Kompresi Nyata Pada implementasi tipikal PI System dengan Swingdoor: data analog yang masuk Snapshot setelah Exception (misal 5,000 events/hari per tag) akan terkompresi menjadi 200–500 events/hari di Archive — rasio kompresi 10:1 hingga 25:1 tergantung "kebisingan" sensor. Untuk Wayang Windu dengan 443 tag aktif, ini berarti penghematan storage yang sangat signifikan dari ukuran archive 10,240 MB.

Swingdoor vs Step Compression

〰️
Swingdoor (step=Off)

Untuk sensor analog — interpolasi LINEAR antar titik tersimpan. Digunakan untuk: tekanan (A1_PIT), temperatur (A1_TE), vibrasi (A1_YIY), flow (FIT). Semua tag ABB dan SETPOINT di Wayang Windu.

⬛
Step Compression (step=On)

Untuk tag digital/status — interpolasi STEP (nilai konstan hingga berubah). Digunakan untuk: motor status (Running/Stopped), valve status (Open/Closed), alarm states. Nilai tidak pernah "di antara dua state".

Life Cycle Archive

1

Archive Baru Dibuat (New Primary Archive)

Saat archive aktif hampir penuh atau secara terjadwal, PI membuat archive baru. Di Wayang Windu ukuran per file = 10,240 MB. Archive lama menjadi "read-only" dan tetap bisa di-query.

2

Data Ditulis ke Archive Primer (Primary Write)

piarchss.exe menulis data dari Swingdoor compression ke archive aktif secara berurutan berdasarkan timestamp. Proses ini berlangsung terus menerus selama PI Server running.

3

Online Reprocessing (Backfill)

Data yang datang terlambat (late arriving data, misal dari buffering saat PI Server sempat down) di-reprocess dan disisipkan ke archive sesuai timestamp-nya. Fitur ini terlihat di PI SMT sebagai "Online reprocessing".

4

Backup Archive (Incremental)

Setiap hari pukul 03:15 AM, PI menjalankan backup INCREMENTAL — hanya data baru sejak backup terakhir yang di-copy. Ini lebih efisien dari full backup. Di Wayang Windu sudah konfirm berjalan (backup history terlihat 3 record sukses).

5

Archive Swap (Auto atau Manual)

Saat archive mencapai 100% penuh, PI otomatis swap ke archive baru (jika sudah disiapkan). Untuk Wayang Windu archive saat ini baru 5.9% — estimasi masih bisa menampung data bertahun-tahun.

07 — Troubleshooting

Troubleshooting Umum

Tag Muncul di Snapshot tapi Tidak di Archive

🔍
Penyebab Umum
  • archiving = 0 — archiving dimatikan untuk tag tersebut
  • compmax sangat besar dan nilai sangat stabil — Swingdoor belum trigger
  • piarchss.exe tidak berjalan — cek PI Services di PI SMT
  • Archive penuh dan belum ada archive baru yang disiapkan

A1_PIT0620, 1
@endt

Data "Hilang" — Gap di Trend PI Vision

⚠️
Kemungkinan Penyebab
  • Interface OPCInt atau SETPOINT Connector stop → tidak ada data masuk ke Snapshot
  • PI Server down → PIBufss.exe menyimpan data sementara, akan di-backfill saat PI kembali online
  • ExcDev terlalu besar → terlalu banyak data di-drop, trend terlihat "kasar"
  • CompDev terlalu besar → detail trend hilang, hanya titik ekstrem yang tersimpan

Snapshot Menunjukkan "No Data" atau "Pt Created"

1

Cek Interface Status

PI SMT → IDWWDW25SCADA01 → Interfaces → pastikan OPCInt running dan connected ke 172.18.1.37.

2

Cek PI Trust

PI SMT → Security → Trusts → pastikan Trust "ABB OPC FES" dengan IP 172.18.1.37 dalam status Enabled = True.

3

Verify Attribute Tag

Cek pointsource = ABB-OPC, instrumenttag = exact OPC Item ID (case-sensitive), Location1=1, Location3=1.

4

Cek State and Bad Points

PI SMT → Data → State and Bad Points → lihat apakah tag masuk kategori Comm Fail, Shutdown, atau System. Ini petunjuk root cause.
